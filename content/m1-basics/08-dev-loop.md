## 用 Claude Code 开发的完整工作流是什么？

**最高效的 Claude Code 开发循环是五个阶段：规划 → 实现 → 审查 → 测试 → 提交。** 你负责规划和审查（需要判断力），AI 负责实现、测试和提交（体力活）。

```
  规划 → 实现 → 审查 → 测试 → 提交
  (你)    (AI)    (你)    (AI)    (AI)
```

下面用一个具体的例子——给用户模块添加头像上传功能——走一遍完整流程。

## 阶段 1：如何让 Claude Code 先分析再动手？

**在 Prompt 末尾加一句"暂时不要写代码，只做分析"。** 这一步最容易被跳过，也最容易省下后面的返工时间。

```bash
$ "我要给用户模块添加头像上传功能。
   请先分析：
   1. src/modules/user/ 目前的代码结构
   2. 有没有现成的文件上传逻辑
   3. 需要修改哪些文件
   暂时不要写代码，只做分析"
```

如果不加最后那句，Claude Code 很可能直接跳到写代码阶段——它很积极，但有时候积极过了头。规划阶段还有一个好处：Claude Code 经常能帮你发现自己没想到的问题，比如项目里其实已经有一个废弃的上传模块可以复用。

## 阶段 2：实现时为什么要拆成小步？

**一次只做一个明确的任务，做完确认没问题再继续。** 如果一口气让 Claude Code 把所有东西都写完，出了问题很难定位是哪一步错了。

```bash
# 第一步：类型定义
$ "在 User 接口添加 avatarUrl 字段"

# 审查确认后...

# 第二步：上传服务
$ "创建 uploadAvatar 函数..."

# 第三步：路由
$ "添加 PUT /api/users/:id/avatar 路由"
```

分步做的好处：每一步改动都很小，review 起来很快，发现问题也容易回退。

## 阶段 3：审查时重点看什么？

**每一步实现之后重点关注三个方面：**

- **逻辑正确性**：有没有边界情况没处理？
- **风格一致性**：是否和项目现有代码一致？
- **复杂度控制**：有没有过度设计？Claude Code 有时候会写出比你需要的更复杂的代码

如果发现问题，直接在当前对话里指出来让它改。比 accept 之后再开新对话修要高效得多。

## 阶段 4：如何让 Claude Code 写出高质量测试？

**在 Prompt 里明确列出要覆盖的测试场景。** 不列的话，Claude Code 通常只会写一两个 happy path 测试，边界情况容易漏掉。

```bash
$ "为 uploadAvatar 写单元测试：
   覆盖：成功上传、文件过大、格式不支持、用户不存在"
```

## 阶段 5：如何用管道自动生成 Commit Message？

**用 `git diff --staged | claude --print` 组合，让 Claude Code 根据实际 diff 生成 commit message：**

```bash
$ git diff --staged | claude --print "生成 commit message"
```

比自己想措辞快多了，而且它能看到所有改动，不会遗漏。
