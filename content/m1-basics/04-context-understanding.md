## 什么是 Claude Code 的上下文窗口？

**上下文窗口是 Claude Code 单次对话中能「看到」的信息总量，大约 200K token。** 粗略换算：1 个中文汉字约 1-2 个 token，1 行代码约 10-20 个 token。

上下文里装的不只是你敲进去的文字，还包括：

- 之前的对话历史
- 读取过的文件内容
- 系统指令
- CLAUDE.md 项目上下文

你可能遇到过这种情况：聊着聊着，Claude Code 的回答质量明显下降了，感觉它「变笨了」。这不是错觉——对话越长，上下文中的「噪声」越多，真正有用的信息占比就越低。

## 上下文的 Token 是怎么被消耗的？

**对话每多一轮，输入 Token 就累积增长。十轮对话后，一半以上的上下文窗口就被占满了。**

| 内容 | 约 Token 数 | 占比 |
|------|------------|------|
| 系统指令 + CLAUDE.md | ~5,000 | 2.5% |
| 你的第一轮对话 | ~500 | 0.25% |
| Claude 读取了 3 个文件 | ~15,000 | 7.5% |
| Claude 的第一轮回复 | ~3,000 | 1.5% |
| ...继续 10 轮... | ~100,000+ | 50%+ |

这还不算它中途读取的其他文件。所以「省着用」不是一句空话。

## 如何有效管理 Claude Code 的上下文？

掌握三条策略，就能让上下文始终高效运转：

### 策略一：新任务，新对话

**这是最简单也最有效的一条。** 当你要开始一个不相关的任务时，别在老对话里继续——直接开新的。之前对话里积累的上下文对新任务来说全是噪声。

### 策略二：精确引用文件

**不要让 Claude Code「看看我整个 src 目录」。** 每读一个文件就是在消耗上下文，所以指定具体文件路径，只让它看它需要看的东西。

给 Claude Code 的指令越精确，它消耗的上下文越少，回答质量反而越高。这和给人布置任务其实是一个道理——范围越模糊，对方要做的「理解工作」就越多。

### 策略三：及时总结，重新开始

如果一个对话确实需要持续很多轮，可以在中途让 Claude Code 总结一下当前进展，然后把总结复制到新对话里继续。这相当于手动做了一次「上下文压缩」。

> 把上下文当成稀缺资源来管理。每一条消息、每一个被读取的文件都在消耗它。像管理内存一样管理上下文——这个类比比你想象的更贴切。
